   - IMPORTANTE: a chamada de refresh usa `axios.post` puro (import direto do axios), NAO `api.post`, para nao disparar o interceptor na propria chamada de refresh

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage -> API | Tokens JWT armazenados no cliente atravessam para o backend via header Authorization |
| API response -> localStorage | Tokens recebidos do backend sao armazenados no cliente sem verificacao de assinatura |
| User input -> API | Credenciais de login (email/senha) enviadas ao backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | localStorage tokens | accept | Decisao D-02 lockada — painel admin-only, single user. Mitigacao adicional via CSP no servidor (fora do escopo frontend) |
| T-04-02 | Elevation of Privilege | Refresh token replay | mitigate | Backend tem ROTATE_REFRESH_TOKENS=True e BLACKLIST_AFTER_ROTATION=True — interceptor salva novo refresh token apos cada uso |
| T-04-03 | Tampering | Redirect apos login | mitigate | Redirecionar sempre para `/` hardcoded — nunca ler URL de query param ou referrer |
| T-04-04 | Denial of Service | Loop infinito no interceptor | mitigate | Flag `_retry` na requisicao original impede tentativa recursiva de refresh |
| T-04-05 | Information Disclosure | Token em console.log | mitigate | Nenhum console.log de tokens no codigo — interceptor nao loga headers |
| T-04-06 | Spoofing | Credenciais em plaintext | accept | HTTPS em producao (servidor); em dev, localhost sem HTTPS e aceitavel |
</threat_model>
